# Executive PPT 会话内长任务恢复与进度可见性设计

- 日期：2026-07-08
- 状态：Draft（待用户复审）
- 范围：`executive-ppt` 会话内的 PPT 预览生成等待态、刷新恢复、未读提示
- 路线：方案 B —— 会话内任务卡 + 后台任务轮询 + 左侧会话摘要提示

## 1. 背景与问题

当前 executive-ppt 的预览生成体验，本质上仍是“同步等待 + 轮播 loading 文案”：

- 前端能显示若干 `loadingMessages`，但这些文案不是任务真实阶段。
- 用户在等待时看不到“任务是否还活着、现在做到哪一步、预计还要多久”。
- 刷新页面或离开后再回来，无法可靠恢复到同一个生成任务的进度。
- 左侧会话列表没有明确提示“哪个会话刚刚返回了新的结果/事件”，用户容易漏掉已完成的任务。

问题不在于“缺少一个更花哨的 loading 动画”，而在于：

1. 长任务没有成为会话内的第一等对象。
2. 会话状态仍偏向 `latestPreview/latestExport` 这样的单例聚合，不适合一个会话内并行多个 PPT 预览任务。
3. 未读提示和任务状态没有形成清晰边界。

## 2. 目标与非目标

**目标**

- 让 PPT 预览生成从“像卡死的同步等待”升级为“会话内可恢复的后台任务”。
- 用户刷新页面 / 重新进入链接后，可以恢复到同一个会话里正在进行或刚完成的任务。
- 同一会话允许并行多个 PPT 预览任务，状态不相互覆盖。
- 左侧会话列表明确提示：
  - 哪些会话有新的正文消息或事件未读；
  - 哪些会话仍有运行中的 PPT 任务。
- 不新增独立任务页或额外选项卡，复用现有会话 UI。

**非目标**

- 第一版不引入 SSE / WebSocket / 长连接。
- 第一版不提供“取消生成”“重试生成”按钮。
- 第一版不做全局任务中心。
- 第一版不做“边生成边逐步展示单个预览方向”的渐进结果流。
- 第一版不改 PPT 生成核心算法，只改任务承载、状态同步和 UI 呈现。

## 3. 方案选择

本轮讨论过三类路径：

### 方案 A：粗粒度后台任务轮询

把同步生成改成后台任务，前端只轮询 `queued/running/succeeded/failed`。

**优点**

- 实现成本最低。
- 能较快缓解“像卡死”的问题。

**缺点**

- 用户仍然不知道任务卡在哪一步。
- 只能证明“任务没死”，不能有效管理预期。

### 方案 B：阶段化任务卡 + 会话恢复

后台任务带真实阶段和心跳；会话流里插入可持续更新的任务卡；刷新后可恢复；左侧列表按消息/事件给红点，按任务摘要给运行中标记。

**优点**

- 预期管理和复杂度最平衡。
- 完整覆盖刷新恢复、会话未读、并行任务。
- 基本不改变用户心智。

**缺点**

- 需要把 PPT 状态从单例聚合升级为任务列表模型。

### 方案 C：阶段化任务卡 + 渐进结果展示

在方案 B 上，结果不必等全部完成后一次出现，而是先出大纲、再出部分方向、最后补齐。

**优点**

- 体感最佳。

**缺点**

- 状态机复杂度明显升高。
- 很容易把第一版做成过重系统。

### 结论

采用 **方案 B**，并给未来升级到方案 C 预留数据结构与接口空间。

## 4. 核心产品形态

### 4.1 不新增选项卡

不引入单独的“任务”选项卡、任务页或右侧独立面板。

原因：

- 任务进度不是独立工作台，而是当前会话的一部分。
- 把消息和任务拆到两个入口，会增加认知负担，并放大会话红点与右侧内容不同步的问题。
- 用户心智应该是“去哪个会话发起，就回哪个会话看结果”。

### 4.2 右侧会话区

右侧会话区只增加两类内容：

1. **任务卡消息**
   - 放在触发 PPT 生成的那轮交互后面。
   - 外观像一条 system / assistant 卡片。
   - 同一张卡在轮询过程中持续更新，不新增消息。

2. **关键节点事件消息**
   - 在关键节点额外落库为可读消息/事件。
   - 负责未读红点和历史可追溯性。

### 4.3 左侧会话列表

左侧列表只增加两个轻量状态：

- `小红点`
  - 代表该会话有新的正文消息或事件未读。
- `运行中标记`
  - 代表该会话里至少存在一个运行中的 PPT 任务。
  - UI 形态可以是小 spinner 或轻量 badge。

左侧**不展示**具体任务阶段，不承载细节。

## 5. 会话内任务卡设计

### 5.1 任务卡生命周期

每次用户发起 PPT 预览生成，都创建一张独立任务卡。

单张任务卡生命周期：

- `queued`：已排队
- `running`：执行中，阶段持续更新
- `succeeded`：完成
- `failed`：失败

任务完成或失败后，任务卡保留在会话流中，不自动消失。

### 5.2 任务卡展示字段

任务卡建议只展示高价值、低噪音信息：

- 任务标题
  - 例如：`正在生成 PPT 预览`
- 当前阶段文案
  - 例如：`正在生成第 2/4 个方向`
- 简短解释
  - 例如：`已确认 brief，正在渲染预览页面`
- 状态标签
  - `排队中 / 运行中 / 已完成 / 失败`
- 已耗时
- 最近更新时间
- 粗粒度进度

### 5.3 任务卡不提供的内容

第一版任务卡**不提供**：

- 取消按钮
- 重试按钮
- 技术日志明细
- 复杂操作区
- 独立分页切换

### 5.4 多任务并行

同一会话允许并行多个 PPT 任务，因此：

- 任务卡不是“会话唯一卡片”。
- 每次生成都插入一张新的任务卡。
- 多张卡按发起顺序稳定存在。
- 同一会话的任务互不覆盖。

## 6. 状态机

### 6.1 任务总状态

第一版只使用四个总状态：

- `queued`
- `running`
- `succeeded`
- `failed`

### 6.2 阶段状态

建议先使用五个粗粒度阶段：

- `brief_validating`
- `story_planning`
- `variant_generating`
- `preview_rendering`
- `session_persisting`

### 6.3 阶段文案

示例映射：

- `queued + brief_validating`
  - `已排队，准备校验需求`
- `running + story_planning`
  - `正在规划结构与故事线`
- `running + variant_generating`
  - `正在生成第 2/4 个方向`
- `running + preview_rendering`
  - `正在渲染预览`
- `running + session_persisting`
  - `正在保存预览会话`
- `succeeded`
  - `预览已生成`
- `failed`
  - `生成失败`

### 6.4 粗进度

不追求真实百分比，采用离散阶段进度即可：

- `0/5` brief 校验
- `1/5` 结构规划
- `2/5` 生成方向
- `3/5` 渲染预览
- `4/5` 写入 session
- `5/5` 完成

目标是“可信”，不是“绝对精确”。

## 7. 数据模型

核心原则：`消息`、`任务`、`会话摘要` 三层分开。

### 7.1 会话消息

继续使用现有会话消息体系。

用途：

- 展示对话历史
- 作为未读红点依据
- 承载关键节点事件消息

### 7.2 PPT 任务记录

每条任务需要显式绑定：

- `taskRunId`
- `conversationId`
- `agentId`
- `taskType = ppt_preview`

建议字段：

- `taskRunId`
- `conversationId`
- `agentId`
- `taskType`
- `requestSummary`
- `status`
- `stage`
- `stageLabel`
- `progressCurrent`
- `progressTotal`
- `lastHeartbeatAt`
- `startedAt`
- `updatedAt`
- `finishedAt`
- `previewSessionId`
- `resultSummary`
- `errorCode`
- `errorMessage`

### 7.3 会话摘要字段

会话列表只需要轻量聚合字段：

- `hasUnread`
- `unreadCount`
- `hasRunningPptTask`
- `runningPptTaskCount`
- `lastMessageAt`
- `lastTaskEventAt`

### 7.4 现有单例聚合态的问题

当前 `conversation_state` 偏向：

- `latestPreview`
- `latestExport`
- `phase`

这类单例态不适合“一个会话内多个并行 PPT 任务”。

因此第一版建议调整为：

- `conversation_state` 继续保留，作为顶部轻摘要；
- `taskRuns[]` 成为 PPT 任务事实来源；
- 多任务一律以 `taskRunId` 作为稳定主键。

## 8. 消息与事件写入规则

### 8.1 原则

- 中间阶段推进：只更新任务卡，不写新消息。
- 关键节点：写正文/事件消息。

### 8.2 必须落消息的关键节点

建议落库为可读消息/事件的节点：

- `任务已开始，正在生成 PPT 预览`
- `PPT 预览已生成，可查看`
- `本次预览已降级生成`
- `PPT 预览生成失败`

### 8.3 不落消息的节点

以下只更新任务卡，不单独刷消息：

- `正在规划结构`
- `正在生成第 1/4 个方向`
- `正在生成第 2/4 个方向`
- `正在渲染预览`
- `心跳更新时间变化`

### 8.4 设计收益

这样可以同时满足：

- 会话流不被中间态刷屏；
- 左侧红点来源清晰；
- 用户能回溯“什么时候开始、什么时候结束、有没有失败”。

## 9. 刷新恢复与重新进入

### 9.1 恢复主键

任务恢复基于：

- `conversationId`
- `taskRunId`
- `agentId`
- `taskType = ppt_preview`

不是基于页面本地 state。

### 9.2 恢复流程

用户刷新页面 / 重新进入会话时：

1. 拉取会话详情；
2. 返回 `messages[] + taskRuns[] + conversationState`；
3. 先渲染历史消息；
4. 再把任务卡按 `taskRunId` 恢复到会话流中；
5. 若存在 `queued/running` 任务，则继续轮询这些任务。

### 9.3 多任务恢复

同一会话允许多个并行任务，因此恢复时：

- 不能只看 `latestPreview/latestExport`；
- 必须返回 `taskRuns[]`；
- 最近运行中的任务优先显示；
- 最近完成的任务随后显示。

## 10. 轮询策略

### 10.1 不使用长连接

第一版不建立长连接。

原因：

- 当前需求是后台长任务状态同步，不是高频双向实时协作。
- 短轮询更稳定、便宜、易恢复。
- 先做轮询，可以较低成本验证交互形态。

### 10.2 当前会话任务轮询

当前打开的会话，轮询运行中的 `taskRunId[]`。

建议频率：

- 页面聚焦：`2.5s`
- 页面失焦：`8s`
- 页面重新聚焦：立即补拉一次
- 全部任务终态：停止轮询

### 10.3 左侧会话摘要轮询

左侧只轮询会话摘要，不轮询任务明细。

建议频率：

- 页面聚焦：`10s`
- 页面失焦：`20-30s`

### 10.4 超时策略

若 `lastHeartbeatAt` 超过阈值：

- 前端展示 `耗时较长，仍在处理中`
- 不自动判失败

只有后端明确写入 `failed`，前端才显示失败态。

## 11. API 设计

### 11.1 会话详情接口

用途：会话首次进入、刷新恢复。

返回：

- `conversation`
- `messages[]`
- `taskRuns[]`
- `conversationState`

约束：

- `taskRuns[]` 只返回最近任务，例如最近 10 个，避免详情过重。

### 11.2 任务状态批量接口

用途：当前会话内进行中任务的高频轮询。

入参：

```json
{
  "taskRunIds": ["101", "102", "108"]
}
```

返回：

```json
{
  "data": [
    {
      "taskRunId": "101",
      "conversationId": "conv_1",
      "status": "running",
      "stage": "variant_generating",
      "stageLabel": "正在生成第 2/4 个方向",
      "progressCurrent": 2,
      "progressTotal": 5,
      "updatedAt": 1780000000,
      "lastHeartbeatAt": 1780000000,
      "previewSessionId": null,
      "errorCode": null,
      "errorMessage": null
    }
  ]
}
```

### 11.3 会话列表摘要接口

用途：左侧列表轮询。

每个会话额外返回：

- `hasUnread`
- `unreadCount`
- `hasRunningPptTask`
- `runningPptTaskCount`

## 12. 未读红点规则

### 12.1 红点依据

红点只基于：

- 新的正文消息
- 新的关键事件消息

### 12.2 红点不依据

红点**不基于**：

- 任务心跳
- 任务卡字段更新
- 中间阶段推进

### 12.3 设计原因

这样能保证：

- 红点含义稳定：有新内容可读；
- 不会因为纯后台心跳而误点亮；
- 用户点进会话后，能看懂“为什么有红点”。

## 13. 前端渲染规则

### 13.1 右侧会话流

- 每次发起 PPT 生成，插入一张任务卡。
- 任务卡按 `taskRunId` 稳定渲染，不因轮询而重排。
- 同一会话多任务并行时，按发起顺序显示多张卡。
- 完成后任务卡保留为完成态。
- 若任务成功，现有预览组件直接接管显示结果。

### 13.2 左侧会话列表

- 红点：有未读正文/事件；
- spinner / badge：该会话存在运行中的 PPT 任务；
- 若并行任务大于 1，可显示轻量数字 badge。

左侧不展示阶段细节。

## 14. MVP 范围

第一版必须包含：

- 会话内任务卡
- 会话详情返回 `taskRuns[]`
- 批量任务状态轮询接口
- 左侧会话列表的未读红点与运行中标记
- 关键节点事件消息
- 刷新恢复

第一版不包含：

- SSE / WebSocket
- 取消任务
- 重试任务
- 渐进展示部分预览结果
- 全局任务中心
- 精确 ETA 预测

## 15. 实施顺序

建议顺序：

1. 后端补充 `taskRuns[]` 模型与轮询接口；
2. 会话详情支持刷新恢复；
3. 前端在会话流中渲染单张任务卡；
4. 支持同一会话多任务并行；
5. 左侧列表增加红点与运行中标记；
6. 最后补齐文案与超时体验优化。

## 16. 风险与对策

### 风险 1：单例聚合态覆盖多任务

**问题**

当前 `latestPreview/latestExport` 思路会让多任务互相覆盖。

**对策**

以 `taskRuns[]` 为事实来源，`conversation_state` 只保留轻摘要。

### 风险 2：任务卡恢复位置不稳定

**问题**

刷新后若只按时间排序，任务卡容易漂移。

**对策**

任务卡需绑定到其发起轮次的会话上下文，恢复时按稳定锚点插入。

### 风险 3：未读规则被任务心跳污染

**问题**

若把任务字段更新也计入未读，左侧红点会混乱。

**对策**

只把正文/关键事件消息纳入未读计算。

## 17. 未来扩展

本设计为后续升级预留空间：

- 可在后续升级到 SSE，而不改前端任务卡消费模型；
- 可在第二期加入渐进式预览结果；
- 可在更远期加入全局任务中心，但不影响当前“会话内任务”主心智。
