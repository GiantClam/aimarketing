# 高频业务卡片信息压缩设计

日期：2026-06-21  
状态：已确认，可进入实施规划  
Owner：Codex

## 目标

为 `aimarketing` 建立一套统一的高频业务卡片压缩规范，解决当前卡片展示信息过多、主次不分、同屏扫描效率低的问题。

这次设计只处理高频业务卡片，不处理整站所有卡片，也不重做整体视觉风格。

本次设计目标：

- 让用户可以在 3 秒内扫完整屏卡片
- 让卡片只承担“识别、判断、进入”三件事
- 统一文字卡片和图文卡片的信息骨架
- 降低卡片高度波动与信息噪音
- 把详情信息移出卡面，回到详情页、页头总览或次级菜单

## 背景与问题

当前项目里的高频业务卡片普遍存在同一类问题：

- 同时展示标题、长摘要、多个 badge、多个统计、多个按钮
- 卡片把“索引入口”和“详情说明”混在一起
- 同一屏的卡片高度差异明显，阅读节奏被打断
- 图文卡片除了封面，还叠了第二套文字结构，信息重复
- 用户在点击前需要读太多内容，难以快速判断“这是不是我要的”

这类问题在以下高频区域表现最明显：

- 工作流卡片
- tools 卡片
- knowledge 卡片
- workspace / template / capability 卡片

这些区域都承担入口型导航任务，因此卡片必须优先服务“快速决策”，而不是“就地读完”。

## 设计结论

本次采用统一的 `A` 规则：

> 每张高频业务卡片只保留：`标题 + 1 行摘要 + 1 个状态/标签 + 1 个主操作`

这是一条硬规范，不按卡片类型放宽，不保留“每类卡片自己决定”的自由度。

## 核心原则

1. **卡片是索引，不是详情页**
   卡片只回答“这是什么”和“要不要点进去”，不负责完整介绍。

2. **信息骨架统一**
   无论是纯文字卡片还是图文卡片，都必须收敛到同一信息结构。

3. **主次关系单一**
   每张卡片只能有一个状态信号、一个主操作，不能并列争抢注意力。

4. **摘要只保留一句**
   摘要只回答用途，不解释流程，不列能力清单，不写 proof points。

5. **详情信息回流到详情层**
   统计、时间、作者、补充标签、次级操作都应离开卡面。

6. **卡片高度稳定优先**
   同屏卡片应尽量保持一致高度，避免因为信息多少造成断裂式排版。

## 统一卡片规范

### 卡面允许出现的元素

每张高频业务卡片固定只允许出现 4 个核心元素：

1. 标题
2. 1 行摘要
3. 1 个状态或标签
4. 1 个主操作

### 标题规则

- 标题允许 `1-2` 行
- 标题优先保证业务名称完整
- 不再叠加副标题
- 不在标题下再拼额外的技术标识

### 摘要规则

- 摘要严格限制为 `1` 行
- 超出部分全部截断
- 摘要只回答“这是什么 / 做什么”
- 禁止写流程说明、能力列表、场景列表、营销文案段落

### 状态 / 标签规则

- 每张卡片只允许出现 `1` 个状态或标签
- 状态优先于分类标签
- 当状态存在时，不再并排显示多个 badge
- 允许的状态示例：`Live`、`Archived`、`Available`、`Processing`、`Enterprise`

### 主操作规则

- 每张卡片只允许出现 `1` 个主操作
- 主操作文案必须是入口动作，例如：`打开`、`查看`、`运行`、`预览`
- 卡面不再并列展示第二按钮
- 次级动作必须移出卡面

### 图文卡片规则

- 图片只作为封面、识别锚点或氛围辅助
- 图片不再承载第二套信息结构
- 不在图片区域叠加大量文案、统计、标签组合
- 图文卡片的正文信息结构仍然遵守 `标题 + 1 行摘要 + 1 状态 + 1 操作`

## 必须移出卡面的信息

以下信息默认不再保留在卡面中：

- 更新时间
- 作者信息
- 数量统计
- proof points
- 多组标签
- scene / media / access 等并列 badge
- 长描述
- 节点数、连线数、chunk 数等结构性统计
- 双按钮或多按钮操作
- 大段辅助说明

这些信息应迁移到以下位置之一：

- 详情页
- 页面顶部总览区
- 卡片外部列表工具栏
- hover / kebab menu / 二级菜单

## 各类卡片映射规则

### 1. Workflow 卡片

保留：

- 标题
- 1 行用途摘要
- 状态
- `打开`

移除：

- 更新时间
- 节点数
- 连线数
- `复制`

补充规则：

- 若没有 `description`，则使用 `slug` 的人话化版本补摘要
- workflow 卡片不再展示结构统计

### 2. Tool 卡片

保留：

- 标题
- 1 行用途摘要
- 可用性状态
- `打开` 或 `预览`

移除：

- media badge
- access badge
- scene badges
- proof points
- 长说明

补充规则：

- 摘要只回答“这个工具解决什么问题”
- 卡面不再展开适用场景细分

### 3. Knowledge 卡片

保留：

- 标题
- 1 行来源或内容摘要
- 处理状态
- `查看`

移除：

- 更新时间
- chunk 数
- dataset 归属说明
- 附加说明块

补充规则：

- 文档统计保留在页头总览，不回流到单卡
- 卡面只表达“这是哪份知识资产、现在处于什么状态”

### 4. Workspace / Template / Capability 卡片

保留：

- 标题
- 1 行价值摘要
- 状态或唯一标签
- `进入`

移除：

- 多按钮
- 功能清单
- 多 badge
- 大段说明

补充规则：

- 如果当前卡片同时承担“介绍 + 导航”，优先保留导航
- 介绍只允许压缩成一句话

## 信息迁移策略

信息压缩不等于信息丢失。迁移策略如下：

- `状态` 保留在卡面
- `统计` 迁移到页头总览或详情页
- `次级操作` 迁移到详情页或菜单
- `能力细分` 迁移到详情页
- `补充说明` 迁移到详情页或卡片外说明区

这样可以保证：

- 卡片足够轻
- 详情仍可完整
- 不破坏现有业务功能

## 视觉与交互约束

本次不是视觉重做，但要统一以下约束：

- 同一屏卡片高度尽量一致
- 摘要区不因内容长短产生大幅跳动
- 状态 badge 位置固定
- 主操作位置固定
- 图文卡封面高度与文字区比例稳定
- hover 效果可保留，但不能靠 hover 才看懂卡片

## 实施范围

### In Scope

- `components/workflows/workflow-list-page.tsx`
- `components/lead-tools/tool-card-grid.tsx`
- `components/knowledge/knowledge-workspace.tsx`
- 1 到 2 处高频 workspace / template / capability 卡片实现
- 必要的共用卡片信息骨架抽象
- 必要的截断、badge、操作位统一

### Out of Scope

- 全站所有卡片一次性重构
- 整体品牌视觉改版
- 卡片动画体系重做
- 详情页信息改写
- 信息架构大重组

## 推荐实施方式

### 方案选择

本次采用“硬规范卡片”方案，不采用“展开副层”或“按类型自由发挥”。

原因：

- 最能从根本上遏制信息继续膨胀
- 最容易跨页面统一
- 最符合入口型卡片的角色
- 实施范围可控

### 实施步骤

1. 抽取轻量级共用卡片信息骨架
   - 目标是统一槽位，而不是新造复杂组件体系
   - 优先复用现有卡片容器和视觉风格

2. 逐个替换高频业务卡片的信息结构
   - workflow
   - tools
   - knowledge
   - workspace / template / capability

3. 将次级信息迁出卡面
   - 放到详情页、菜单或页头总览

4. 统一截断和布局规则
   - 标题 `1-2` 行
   - 摘要 `1` 行
   - 1 个 badge
   - 1 个主操作

## 风险与处理

### 风险 1：用户感觉信息“少了”

处理：

- 不删除信息，只迁移位置
- 把详情保留在详情页或菜单

### 风险 2：不同卡片类型认为自己特殊

处理：

- 统一按入口型卡片处理
- 只有详情层可以特殊，卡面不特殊

### 风险 3：局部页面仍偷偷回填信息

处理：

- 用统一槽位限制卡面内容
- 在 review 时按硬规则检查

## 验收标准

改造完成后应满足：

- 用户可在 3 秒内扫完整屏卡片
- 同屏卡片高度明显更整齐
- 每张卡片只出现 1 个主操作
- 不再出现 3 个以上 badge
- 不再出现 2 段以上正文
- 不再在卡面同时出现“长摘要 + 多统计 + 多按钮”
- workflow / tools / knowledge / workspace 卡片都遵守同一骨架

## 不做的事

本次设计明确不做：

- 把所有卡片都改成完全同一个视觉组件
- 让卡片承担详情页职责
- 通过 hover 补回大量信息
- 重新设计整站视觉语言

## 最终结论

`aimarketing` 的高频业务卡片应统一收敛为：

> `标题 + 1 行摘要 + 1 个状态/标签 + 1 个主操作`

这是一条产品级展示规范，而不是单页面优化建议。

后续新增高频业务卡片时，默认必须遵守这条规范，除非有明确的破例理由，并在设计评审中单独说明。
