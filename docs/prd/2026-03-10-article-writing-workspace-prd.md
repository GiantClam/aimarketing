# 文章写作工作台 PRD

**状态**: Approved for implementation  
**日期**: 2026-03-10  
**适用阶段**: 进入实现、E2E 测试、上线回归测试  
**本轮评审结论**:
- 旧“文案写作专家”下线。
- “图文一键复制”采用“文本一键复制 + 图片下载/链接复制”口径。
- X 与 Facebook 同时支持“单篇长文”和“线程/多段帖文”两种输出模式。
- 信息架构采用“统一文章写作入口，平台内选择”，不采用 4 个平台长期并列入口。
- 新功能页面风格需与现有网站一致，并适当增加动效。
- 网站部署在 Vercel，新功能不采用 Python 作为主后端，统一使用 Node.js / Next.js 后端能力实现。

- 当前产品已有“专家 Agent”分类，包含品牌战略顾问、增长顾问，以及独立的“文案写作专家”会话能力。
- 当前顾问会话基于 `/dashboard/advisor/[type]/[[...id]]`、`DifyChatArea`、`AdvisorSidebarItem` 和 Dify 流式接口实现。
- `D:\OpenCode\writer` 已具备可复用的内容链路资产：研究、提纲、图文生成、Markdown 输出、图片生成、微信公众号草稿上传雏形。

## 1. 背景与问题

当前“文案写作专家”仍然是单一顾问形态，无法表达不同平台的图文创作差异，也没有形成完整的“文章写作工作台”体验。用户当前缺少以下能力：

1. 无法在“专家顾问”之外，以独立分类管理图文写作能力。
2. 无法针对不同平台选择专属写手人格、结构模板、图片规格和输出格式。
3. 无法在对话生成后进入结构化编辑状态，对文章做微调、预览和二次加工。
4. 无法把生成结果整理成可直接发布的“图文发布包”。

本次需求要把“文章写作”做成与“专家顾问”并列的一类工作台能力，并首批支持 4 个平台：

1. 公众号文章写手
2. 小红书图文写手
3. X 文章写手
4. Facebook 文章写手

但入口策略调整为：

1. 统一保留一个“文章写作”入口。
2. 用户进入后在工作台内部选择目标平台。
3. X 与 Facebook 再选择输出模式。

---

## 2. 目标

### 2.1 产品目标

构建一个面向多平台内容创作者的“文章写作工作台”，支持用户从对话开始，完成研究、写作、配图、编辑、预览与复制发布的完整闭环。

### 2.2 业务目标

1. 将原本单一的“文案写作专家”升级为更清晰的多平台图文创作能力。
2. 提升用户从“提出需求”到“拿到可发布内容”的效率。
3. 为后续接入直接发布、素材库、内容日历、多平台改写等能力预留架构。

### 2.3 成功标准

1. 用户从统一入口进入后，可在 4 个平台中任选一个发起会话。
2. 单次会话可生成完整图文文章，至少包括正文和 1 张封面图。
3. 用户可在页面内完成 Markdown 编辑、实时预览、继续对话微调。
4. 用户可一键复制对应平台的发布文案，并获取本次生成的图片素材。
5. X 与 Facebook 用户可明确选择输出为“单篇长文”或“线程/多段帖文”。

---

## 3. 非目标

以下内容不纳入本次首版必做范围：

1. 不做真正的自动发布到 X、Facebook、小红书、微信公众号后台。
2. 不做多人协作、评论批注、版本 diff。
3. 不做内容排期、审批流、运营日历。
4. 不做跨平台一键改写矩阵分发。

说明：
- `D:\OpenCode\writer\tools\wechat_draft_uploader.py` 提供了微信公众号草稿上传雏形，但本需求的明确目标是“一键复制发布内容”，不是直接调用平台发布接口。

## 4. 用户与核心场景

### 4.1 目标用户

1. 企业市场人员
2. 创始人/运营负责人
3. 内容编辑与新媒体运营
4. 需要跨平台发布图文内容的个人创作者

### 4.2 核心场景

1. 用户进入“文章写作”后，先选择“公众号”，再生成深度长文与配图。
2. 用户进入“文章写作”后，先选择“小红书”，再生成封面图、短文案、标签和配图说明。
3. 用户进入“文章写作”后，选择“X”，并在“单篇长文”和“线程式文章”之间切换。
4. 用户进入“文章写作”后，选择“Facebook”，并在“单篇长文”和“多段帖文”之间切换。
5. 用户在生成结果基础上继续聊天，让 AI 改标题、补案例、缩长度、换风格。
6. 用户手动在 Markdown 编辑器中调整文案，并通过预览确认排版效果。
7. 用户点击“一键复制”后，把文案复制到目标平台，并下载或复制图片链接完成发布。

## 5. 信息架构与导航

### 5.1 一级结构

侧边栏新增与“专家 Agent”并列的一级分类：

1. 专家 Agent
2. 文章写作

### 5.2 入口策略

不再为 4 个平台在侧边栏长期设置并列子入口，而采用统一入口：

1. 侧边栏仅显示一个 `文章写作` 入口。
2. 点击后进入文章写作首页或默认工作台。
3. 平台选择发生在工作台内部，而不是侧边栏层级。

### 5.3 页面结构

建议采用两级页面：

1. `文章写作首页`
   - 平台选择卡片
   - 最近会话
   - 推荐模板
2. `文章写作工作台`
   - 顶部平台选择器
   - 顶部模式选择器（仅 X / Facebook 显示）
   - 左侧会话历史
   - 中部对话 + 编辑器
   - 右侧预览 + 素材

### 5.4 路由建议

建议新增统一路由而不是每个平台一个顶层路由：

1. `/dashboard/writer`
2. `/dashboard/writer/[sessionId]`

平台与模式通过以下方式表达：

1. URL query 参数
   - `?platform=wechat`
   - `?platform=xiaohongshu`
   - `?platform=x&mode=thread`
   - `?platform=facebook&mode=post-series`
2. 或保存在 session/document 状态中，由进入会话时恢复。

推荐：
- 统一使用 `/dashboard/writer/[sessionId]?platform=...&mode=...`，便于扩展更多平台。

### 5.5 兼容策略

1. 旧 `/dashboard/advisor/copywriting/...` 路由下线，不再作为可见入口保留。
2. “文案写作专家”从“专家 Agent”分组中移除。
3. 旧文案写作会话不迁移到新工作台。
4. 对旧路由访问建议返回 404 或重定向至 `/dashboard/writer`，由实现时定稿。

推荐：
- 首版不做历史文案会话迁移，降低兼容复杂度。

## 7. 功能需求

### 7.1 分类与权限

1. 系统新增“文章写作”侧边栏分组，显示在“专家 Agent”旁边。
2. 侧边栏只显示一个 `文章写作` 入口。
3. 首版权限建议：
   - 后端运行时继续复用 `copywriting_generation`，降低权限系统改造成本。
   - 前端展示文案改为“文章写作”。
4. 后续如需要精细控制，可再拆分：
   - `article_writing_generation`
   - `article_writing_wechat`
   - `article_writing_xiaohongshu`
   - `article_writing_x`
   - `article_writing_facebook`

### 7.2 平台选择与输出规范

工作台必须支持平台内选择，而不是通过侧边栏拆成 4 套长期入口。

每个平台都必须具备独立的：

1. 系统 Prompt
2. 输出结构模板
3. 图片比例与图片数量建议
4. 文案语气
5. 发布格式约束
6. 输出模式切换规则（如适用）

其中 X 与 Facebook 必须支持两种模式：

1. 单篇长文模式
2. 线程/多段帖文模式

模式要求：

1. 用户可在生成前显式选择输出模式。
2. 用户未选择时，系统按平台默认推荐：
   - X 默认推荐“线程式文章”
   - Facebook 默认推荐“单篇长文”
3. 两种模式都必须支持后续微调、预览和复制。

建议首版规范如下：

| 平台 | 内容形态 | 推荐长度 | 图片策略 | 预览重点 |
|---|---|---|---|---|
| 公众号 | 深度长文 | 1500-4000 字 | 封面 16:9，正文配图 2-5 张 | 标题层级、长文排版、图文穿插 |
| 小红书 | 图文笔记 | 200-800 字 | 封面 3:4，配图 4-9 张 | 首屏钩子、分点、标签、移动端阅读 |
| X | 单篇长文 / 线程式文章 | 300-1200 字 | 首图 16:9 或 1:1，配图 1-4 张 | 短段落、强观点、可拆分线程 |
| Facebook | 单篇长文 / 多段帖文 | 400-1500 字 | 首图 1.91:1 或 1:1，配图 1-4 张 | 社区传播感、品牌叙事、CTA |

### 7.3 对话生成

1. 用户进入文章写作工作台后，先选择目标平台，再以对话方式工作。
2. 支持流式生成，保留当前顾问聊天的实时体验。
3. 会话支持历史记录、新建、重命名、删除。
4. 用户可在对话中输入：
   - 选题
   - 目标用户
   - 风格要求
   - 字数要求
   - 是否需要图片
   - 参考资料/链接
   - 输出模式（适用于 X / Facebook）
5. AI 输出必须结构化返回至少以下字段：
   - 标题候选
   - 正文 Markdown
   - 封面图描述或封面图结果
   - 配图列表或配图描述
   - 平台标签/CTA/推荐发布时间等扩展字段
   - 平台
   - 输出模式（单篇长文 / 线程 / 多段帖文）

对于 X 与 Facebook，还需补充结构化字段：

1. `postMode`
2. `postItems`
3. `threadHook`
4. `closingCTA`

### 7.4 完整图文生成

系统必须支持“文字 + 图片”的完整产出，不接受只有纯文本。

最小交付标准：

1. 每篇内容至少 1 张封面图。
2. 公众号内容支持正文内配图。
3. 小红书、X、Facebook 至少返回图片素材建议和可用图片结果。
4. 图片生成失败时，系统必须给出可回退的图片 Prompt，不可静默失败。

### 7.5 Markdown 编辑器

生成完成后，页面进入“编辑工作台”状态，至少包含：

1. Markdown 编辑区
2. 实时预览区
3. 继续对话区
4. 图片素材区

编辑器能力要求：

1. 支持编辑标题、正文、列表、表格、引用、图片链接。
2. 支持插入/替换图片块。
3. 支持恢复 AI 最近一次生成版本。
4. 支持手动修改后继续让 AI 基于当前稿件微调。
5. 支持 X / Facebook 在线程与长文模式间重新生成。

### 7.6 微调能力

微调必须同时支持两种方式：

1. 对话式微调
   - 示例：把标题改得更像公众号爆文
   - 示例：把正文缩短到 600 字
   - 示例：再补 2 个案例
2. 手动编辑式微调
   - 用户直接改 Markdown
   - 再要求 AI 基于当前版本继续改写

系统需明确“AI 基于哪一版内容继续改写”，避免覆盖用户手工修改。

### 7.7 预览

预览至少提供两个层次：

1. Markdown 通用预览
2. 平台化预览

平台化预览要求：

1. 公众号：更接近长文图文排版效果。
2. 小红书：模拟手机端图文笔记效果。
3. X：支持“单篇长文”和“线程式文章”两种预览。
4. Facebook：支持“单篇长文”和“多段帖文”两种预览。

### 7.8 一键复制发布包

用户点击后，应获得可直接用于目标平台发布的内容包。

本期已确认定义为：

1. 一键复制正文文本
2. 一键复制标题
3. 一键复制标签/CTA
4. 一键复制图片 Prompt 或图片说明
5. 批量下载本次图片素材
6. 逐张复制图片链接

技术约束说明：

1. 浏览器对“多张图片 + 富文本 + 纯文本”一次性写入系统剪贴板的支持不稳定。
2. 因此首版不承诺真正把多张图片二进制一起复制进剪贴板。
3. 本期已确认对“图文一键复制”的产品定义为：
   - 文本内容一键复制到剪贴板
   - 图片素材一键下载或逐张复制链接
   - 页面提供平台发布清单

### 7.9 状态持久化

1. 会话级保存用户 prompt、AI 回复、当前稿件、图片素材。
2. 稿件编辑状态可自动保存。
3. 重新进入会话时，恢复：
   - 对话历史
   - 当前 Markdown 稿件
   - 最近预览状态
   - 图片素材列表
   - 当前平台
   - 输出模式

## 8. 页面与交互要求

### 8.1 入口页

`文章写作` 首页应包含：

1. 平台选择卡片
2. 最近会话
3. 常用模板
4. 最近草稿

### 8.2 视觉风格一致性

新功能页面必须延续当前站点既有设计语言，不单独创建新的视觉体系。

具体要求：

1. 继续复用现有 `DashboardLayout`、颜色变量、圆角、边框、阴影、字体和卡片体系。
2. 视觉基调与当前站点保持一致：
   - 浅粉背景
   - 粉色主色
   - 青色强调色
   - 卡片化内容容器
   - 现有按钮、输入框、Badge、Tab 风格
3. 新页面允许在现有风格内做层次优化，但不允许切成全新主题。
4. 平台差异应主要体现在内容结构、图标、预览样式和状态标签上，而不是整体换皮。

### 8.2 工作台页面

建议采用三栏或二栏增强布局：

1. 顶部：平台选择器 + 模式选择器（X / Facebook）
2. 左栏：会话历史
3. 中栏：对话 + 编辑器
4. 右栏：平台预览 + 图片素材

在中小屏上退化为 Tab 切换：

1. 对话
2. 编辑
3. 预览
4. 素材

### 8.3 动效要求

动效目标是增强反馈和层次感，而不是制造花哨感。

必须包含的动效：

1. 页面首次进入时，平台卡片或工作台模块有轻量淡入/上移动效。
2. 平台切换时，编辑区与预览区有短时过渡，避免硬切。
3. 流式生成过程中，对话、状态条、生成按钮有明确的进行中反馈。
4. 复制、下载、应用到编辑器等关键操作有即时成功反馈。
5. 悬停态、选中态、激活态保持与现有站点一致的微动效节奏。

动效约束：

1. 不使用大幅位移、长时间弹跳、夸张缩放。
2. 动效时长建议控制在 150ms 到 300ms。
3. 避免影响编辑器输入、滚动和流式输出性能。
4. 移动端动效需更轻，避免频繁重绘。

### 8.4 关键按钮

1. 开始生成
2. 停止生成
3. 应用到编辑器
4. 基于当前稿件继续润色
5. 恢复 AI 版本
6. 复制标题
7. 复制正文
8. 下载图片
9. 新建会话
10. 切换平台
11. 切换输出模式（X / Facebook）

## 10. 技术方案边界

### 10.1 现有能力可复用

1. Dify / 流式会话接口
2. 会话历史与侧边栏模式

## 13. 风险与已确认事项

### 13.1 主要风险

1. X 与 Facebook 尚无现成 writer Prompt 资产，需要新增平台策略和双模式输出模板。
2. 图片生成耗时较长，容易影响首稿响应速度。
3. “图文一键复制到剪贴板”在浏览器层存在能力边界。
4. 继续使用 `advisor` 模型会让“写作工作台”语义混乱。
5. 旧文案写作入口下线后，需要处理旧链接访问体验与历史会话处置。
6. 统一入口方案对状态管理要求更高，平台、模式、文稿和预览必须同步一致。
7. 若动效实现不当，容易造成编辑区重绘、预览闪烁或移动端卡顿。
8. 若错误地采用 Python 作为主后端，会与 Vercel 当前部署模型冲突，增加交付复杂度。

### 13.2 已确认事项

1. 旧“文案写作专家”下线，不做迁移保留。
2. 首版接受“文本一键复制 + 图片下载/链接复制”作为“图文一键复制”实现口径。
3. X 与 Facebook 首版同时支持“单篇长文”和“线程/多段帖文”两种模式。
4. 首版权限继续复用 `copywriting_generation`，后续再视需要细分。
5. IA 采用“统一入口，平台内选择”，不做 4 个长期并列平台入口。

建议按以下口径进入实现：

1. 新增“文章写作”一级分类，与“专家 Agent”并列。
2. 侧边栏只保留一个 `文章写作` 入口。
3. 进入工作台后，由用户在顶部或首页卡片中选择平台。
4. 下线旧“文案写作专家”入口，统一收口到新的“文章写作”工作台。
5. 复用 `D:\OpenCode\writer` 的 research、图片、平台化 Prompt 思路，不直接复用其 CLI 运行时。
6. 保留现有 Dify 对话基座，但新建 `writer` 页面和文稿状态层。
7. X 与 Facebook 在首版即支持两种输出模式：
   - 单篇长文
   - 线程 / 多段帖文
8. 首版将“一键复制图文内容”定义为：
   - 文本一键复制
   - 图片下载/链接复制
   - 平台化发布清单
9. 新页面在视觉上必须继承现有站点设计语言，并增加轻量、克制的动效反馈。
10. 后端实现统一采用 Node.js / Next.js，Python 资产仅作为迁移参考，不作为线上主后端。

按以上口径，PRD 已可直接进入详细技术设计与实现阶段。
